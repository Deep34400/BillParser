Invoice OCR

A self-hosted tool that turns finance bills (PDFs) into structured, queryable data. Upload a PDF, an extraction engine reads it, and the result is stored in Postgres and shown as a searchable table with full line-item detail.

Problem

Bills arrive as PDFs in many different layouts — every vendor formats invoices differently. Today they are read and re-keyed by hand into spreadsheets or an accounting system. That is slow, error-prone, doesn't scale, and leaves no clean dataset to query for spend analysis, reconciliation, or audit.

Solution

A web app with a single job: ingest bill PDFs and produce trustworthy structured records. Because layouts vary, extraction is provider-agnostic — every engine (Mistral OCR, Azure Document Intelligence, LlamaParse, Textract, Google Document AI) normalizes into one canonical invoice schema. Engines are swapped with a single config value, so accuracy/cost trade-offs can be tuned without touching the app.

One-line pitch: Drop a folder of bills in; get a clean, searchable, exportable table of invoices and line items out — with the extraction engine of your choice behind a stable schema.

Goals & Non-Goals

Goals

  • Extract header fields + line items from arbitrary-layout bill PDFs.
  • Store every bill in Postgres with a stable canonical schema.
  • Make results searchable, reviewable, and exportable (CSV).
  • Keep the extraction engine swappable per cost/accuracy needs.
  • Run fully self-hosted via one docker compose up.
  • De-duplicate re-uploaded files automatically.

Non-Goals (v1)

  • Approval workflows / payments / posting to an ERP.
  • Multi-tenant SaaS, billing, or per-user accounts.
  • Email-inbox ingestion (manual upload only in v1).
  • Editing extracted values in-place (review + re-extract only).
  • Mobile-native app.
  • Training a custom extraction model.

Personas

🧾 Finance Operator

Primary user

Processes incoming bills daily. Wants to stop hand-keying invoices and trust that totals, dates, and line items are captured correctly. Lives in the table view and CSV export.

📊 Finance Manager / Analyst

Consumer of data

Needs clean spend data to reconcile, categorize, and analyze. Cares about completeness, accuracy, and being able to export to a spreadsheet or warehouse.

🛠️ Developer / Admin

Operator of the tool

Deploys and configures it. Chooses the extraction provider, manages keys, monitors failures, and wires new providers or downstream integrations.

UC-1 · Bulk ingest a batch of bills

Actor Finance Operator   Trigger A pile of vendor PDFs to process

Flow: Operator drags 30 PDFs onto the upload zone → each file is hashed, stored, and queued → rows appear as PENDING, move to PROCESSING, then COMPLETED with vendor, totals, and line-item counts populated live.

Success: All non-duplicate files become invoice rows; duplicates are flagged and skipped. No manual keying.

UC-2 · Review a low-confidence extraction

Actor Finance Operator   Trigger Confidence badge looks low or totals seem off

Flow: Operator opens the invoice detail → sees canonical fields, line-item table, confidence score, and the raw OCR text side-reference → compares against the source to confirm or spot a mis-read.

Success: Operator can verify any field against raw OCR output without leaving the app; questionable bills are easy to spot via the confidence badge.

UC-3 · Re-extract or switch engine

Actor Operator / Admin   Trigger A bill extracted poorly on the default engine

Flow: From invoice detail, pick a different provider in the dropdown (e.g. azure for best line-item accuracy) → re-extract → existing line items are replaced atomically with the new result.

Success: Same bill, better result, without re-uploading — and the choice is per-invoice, so engines can be compared head-to-head.

UC-4 · Search and export

Actor Analyst   Trigger Needs spend data for a vendor / period

Flow: Filter the table by status or search by vendor / invoice number / filename → export the (filtered) headers to CSV → open in a spreadsheet or load into a warehouse.

Success: A clean dataset leaves the tool in seconds; no copy-paste from PDFs.

UC-5 · Diagnose & recover from a failure

Actor Admin   Trigger A bill lands in FAILED

Flow: Open the invoice → read the captured error (e.g. missing API key, provider timeout, unreadable scan) → fix config or choose another provider → re-extract.

Success: Failures are explicit and actionable, never silent; recovery is one click after the fix.

Extraction Providers

All engines normalize into one canonical schema. Choose with the EXTRACTION_PROVIDER env var. Pricing/accuracy as of mid-2026 — verify before relying on it.

ProviderKey~Cost / 1k pagesLine itemsPatternStatus
Mistral OCR 3mistral~$2 ($1 batch)strongOCR→markdown + LLM structuringdefault
Azure Document Intelligenceazure~$10best (~93%/87%)prebuilt invoice, structured outbuilt
LlamaParse v2llamaparse~$7.5–12strongOCR→markdown + LLM structuringbuilt
AWS Textract (AnalyzeExpense)textract~$10good (~78%/82%)structured fieldsstub
Google Document AIgoogle~$10–30weaker (~40%)structured fieldsstub
Two extraction patterns. Markdown providers (Mistral, LlamaParse) OCR to markdown, then a second "structuring" LLM (OpenAI / Anthropic / Mistral, via STRUCTURING_MODEL_PROVIDER) maps text → fields. Structured providers (Azure, Textract, Google) return invoice fields directly — no LLM step.
Decision: default to mistral for lowest cost and layout robustness; recommend azure when line-item accuracy matters most. The abstraction means this is a config choice, not a code change.

Frontend

React + Vite, React Router

Upload UI, searchable table, invoice detail with line items, CSV export, live polling while jobs run.

API

Node · Fastify · Prisma

Upload, list/search, detail, re-extract, delete, CSV. Runs extraction fire-and-forget per file.

Storage

Postgres 16

Invoice + LineItem tables, plus raw OCR text and full provider JSON for audit.

Packaged as three containers via Docker Compose (db, api, web). Migrations apply automatically on API start. Files stored on a mounted volume; de-duplicated by SHA-256 of content.

Non-Functional Requirements

Performance

Upload of 50 files accepted without blocking the UI; extraction runs async. A typical 1–3 page bill completes within the provider's round-trip (seconds to ~30s).

Reliability

A failed extraction never blocks others; errors are captured per-invoice and recoverable via re-extract. Duplicate uploads are idempotent (hash dedupe).

Security & Privacy

Self-hosted; bill data and files stay on the operator's infra. Provider API keys via env only. (v1 has no auth — intended for trusted/internal deployment; see roadmap.)

Portability

One docker compose up brings up DB + API + web. No cloud lock-in; provider is swappable by config.

Observability

Structured API logs; per-invoice status, provider, confidence, and raw payloads retained for audit and debugging.

Extensibility

New provider = implement one interface + register it; it appears in the UI automatically. Stable canonical schema shields the rest of the app.
